\section{Implementation, Evaluation, and Results}

\subsection{Introduction}

The previous chapter presented the system design and architecture of the VIP platform. This chapter moves from architectural design to practical implementation, evaluation, and results.

It first describes the implementation of the VIP platform, including the Agent Factory, generated agent runtime, workflow orchestration, Docker-based deployment, and observability integration. It then presents the SLM-Evals project, which was developed to compare LLM and SLM providers according to quality, latency, cost, schema compliance, stability, and domain relevance.

Finally, this chapter presents the evaluation protocol and discusses the obtained results. The objective is to assess whether the implemented solution satisfies the functional and non-functional objectives defined earlier in the report.

\subsection{VIP Platform Implementation}
    \subsubsection{Implemented Use Case and Execution Scenario}
 \label{sec:implemented-use-case}

This scenario was selected because it involves several complementary agent roles and therefore demonstrates the main capabilities of the platform: configurable agent creation, independent containerized execution, schema-based communication, dynamic workflow execution, and runtime observability.

The implemented workflow is composed of four generated agent roles: an extractor, a mapper, a generator, and a sanitizer. In this report, these roles are referred to as \texttt{extractor\_agent}, \texttt{mapper\_agent}, \texttt{generator\_agent}, and \texttt{sanitizer\_agent}. The exact runtime instance names are generated automatically by the platform and do not affect the functional responsibility of each agent.

The workflow starts from a financial PDF or input payload and produces a sanitized financial analysis. The saved application graph is executed as a runtime workflow specification. The detailed technical path used to transform the saved graph into an executable workflow is presented later in Section~\ref{sec:workflow-orchestration-implementation}.

\subsubsection{Agent Factory Implementation}

\label{sec:agent-factory-implementation}

The Agent Factory is the build-time component responsible for transforming an agent creation request into a runnable agent service. In the implementation, this responsibility is handled by the \texttt{builder/} module. The Agent Factory does not execute business logic itself. Instead, it prepares the runtime files, configuration, selected tools, dependencies, and Docker runtime elements required to create an independent generated agent.

The implementation separates assembly logic from runtime logic. The reusable runtime behavior is provided by \texttt{agent\_core/}, while the builder assembles a concrete agent artifact from the selected configuration. This allows several agents to share the same runtime core while behaving differently according to their prompt references, schema references, selected tools, model configuration, and memory settings.

The agent creation process starts when the builder API receives a creation request. This request contains the agent name, provider and model settings, prompt URI, memory configuration, input and output schema URIs, and selected tools. Before generation starts, the request is validated and normalized to prevent incomplete or malformed configurations from being transformed into runtime artifacts.

The main implementation files involved in the Agent Factory are:

\begin{itemize}
    \item \textbf{Builder API} -- \texttt{builder/api/app.py}: exposes the builder FastAPI service.

    \item \textbf{Request validation} -- \texttt{builder/api/schemas/create\_agent\_request.py}: validates the agent creation request.

    \item \textbf{Build coordination} -- \texttt{builder/services/build\_service.py}: coordinates the generation process.

    \item \textbf{Configuration preparation} -- \texttt{builder/services/config\_normalizer.py} and \texttt{builder/services/config\_validator.py}: normalize and validate the received configuration.

    \item \textbf{Runtime artifact assembly} -- \texttt{builder/services/artifact\_assembler.py}: assembles runtime files and writes \texttt{agent\_config.yaml}.

    \item \textbf{Tool and dependency preparation} -- \texttt{builder/services/tool\_selector.py} and \texttt{builder/services/dependency\_resolver.py}: copy selected tools and prepare tailored dependencies.

    \item \textbf{Docker runtime handling} -- \texttt{builder/services/docker\_runtime.py}: builds and starts the generated agent container.

    \item \textbf{Runtime contract exposure} -- \texttt{agent\_core/api/routers/manifest.py}: exposes the runtime \texttt{/manifest} contract.
\end{itemize}

After validation, the builder assembles a controlled runtime artifact from \texttt{agent\_core/} instead of copying the full repository. It then copies only the selected tools, prepares dependency files, writes the generated \texttt{agent\_config.yaml}, builds the Docker image, and starts the generated container. Once the service is running, the \texttt{/manifest} endpoint is used to confirm that the generated agent exposes the expected runtime contract.

    \subsubsection{Generated Agent Runtime Implementation}

\label{sec:generated-agent-runtime-implementation}

After the Agent Factory prepares the runtime package and starts the container, the generated agent becomes an independent FastAPI service. All generated agents share the same reusable \texttt{agent\_core/} scaffold, while their behavior is defined by the generated \texttt{agent\_config.yaml} file. This configuration contains the agent name, provider and model settings, prompt URI, memory configuration, input and output schema URIs, and selected tools.

The generated runtime mainly contains the FastAPI entry point, the reusable agent logic, the generated \texttt{agent\_config.yaml}, API routers for \texttt{/run} and \texttt{/manifest}, selected tools, dependency files, and the Dockerfile. At startup, the service loads its configuration, retrieves the prompt from MLflow, loads the input and output schemas from their \texttt{schemas://} references, and initializes the configured model provider.

Each generated agent exposes two main endpoints. The \texttt{/run} endpoint executes the agent using a standard request envelope composed of a \texttt{trace\_id}, a \texttt{payload}, and optional \texttt{metadata}. The \texttt{/manifest} endpoint exposes the runtime contract of the service, including its identity, version, input schema, and output schema. This endpoint allows the builder and the orchestrator to verify that the running container exposes the expected contract before execution.

Figure~\ref{fig:generated_agent_runtime} summarizes the runtime behavior of a generated agent.

\subsubsection{Validation Layers: Pydantic and JSON Schema}
\label{sec:validation-layers}

The VIP platform applies two complementary validation layers. Pydantic validates API request structures, while JSON Schema validates the business payloads exchanged between agents.

At the API level, Pydantic is used before execution starts. The generated agent runtime validates \texttt{POST /run} requests with \texttt{RunRequest} in \texttt{agent\_core/api/schemas/run.py}. The builder validates agent creation requests with \texttt{CreateAgentRequest} in \texttt{builder/api/schemas/create\_agent\_request.py}. The orchestrator validates dynamic workflow execution requests with \texttt{RunWorkflowSpecRequest} in \texttt{orchestrator/api/schemas/runWorkflowSpec.py}.

After the request envelope is accepted, JSON Schema validates the actual payload. In \texttt{agent\_core/agent.py}, the input payload is checked against the configured input schema, tool arguments are validated before execution, and the final structured result is checked against the configured output schema. At the workflow level, \texttt{orchestrator/services/schema\_service.py} uses \texttt{Draft202012Validator} to validate step inputs and outputs.

The validation flow can be summarized as follows:

\begin{center}
\texttt{Pydantic request validation}
$\rightarrow$
\texttt{JSON Schema input validation}
$\rightarrow$
\texttt{Tool argument validation}
$\rightarrow$
\texttt{Tool/model execution}
$\rightarrow$
\texttt{JSON Schema output validation}
\end{center}

This layered approach prevents malformed requests from entering the system and ensures that each generated agent receives and returns data that respects its configured contract.

    \subsubsection{Workflow Orchestration Implementation}

The workflow orchestration layer executes generated agents in a defined order. In the current implementation, workflows are not loaded from static YAML files. Instead, the workflow created from the user interface is saved as an application graph and transformed into a runtime workflow specification before execution.

When the user launches a saved workflow, the request first reaches the frontend backend through \texttt{POST /orchestration/\{app\_id\}/run}. The backend compiles the saved graph and forwards the runtime specification to the orchestrator through \texttt{POST /run-workflow-spec}. The request is then handled by \texttt{runWorkflowSpec.py} and executed by \texttt{WorkflowExecutionService}.

For the implemented financial workflow, the runtime specification follows the functional order introduced in Section~\ref{sec:implemented-use-case}: extraction, mapping, generation, and sanitization.

Before execution, the orchestrator retrieves the runtime manifest of each target agent using \texttt{ManifestService}. The manifest provides the agent identity, input schema, and output schema. This allows the orchestrator to execute the workflow according to the contract exposed by each generated agent.

For each step, the orchestrator prepares the payload expected by the next agent. The \texttt{AdapterService} aligns the current payload with the target input schema. It first attempts deterministic adaptation, and when this is not sufficient, it relies on the model-backed \texttt{AdapterAgent}.

After the payload is prepared, the orchestrator invokes the target generated agent through \texttt{AgentClient}. The call is performed over the internal Docker network using the runtime endpoint \texttt{http://<agent-container>:8000/run}. The returned result is stored in the workflow execution state and passed to the next step until the final output is produced.

\subsubsection{PostgreSQL Persistence and Table Models}

\label{sec:postgresql-persistence}

PostgreSQL is used as the persistence layer of the VIP platform. It stores platform metadata related to applications, agents, reusable agent templates, and workflow definitions. The persistence layer is implemented with SQLAlchemy and Alembic under the \texttt{frontend/backend/} module, mainly through \texttt{models.py}, \texttt{db.py}, and \texttt{storage/postgres.py}.

The main PostgreSQL tables are:

\begin{itemize}
    \item \textbf{\texttt{apps}} -- stores application or workspace records, including name, description, version, status, creator, and timestamps.

    \item \textbf{\texttt{agents}} -- stores app-scoped agent records, including agent name, type, category, runtime status, container metadata, image tag, endpoints, selected tools, configuration snapshot, and builder request snapshot.

    \item \textbf{\texttt{agent\_library}} -- stores reusable agent templates that can be copied into an application when creating new agents.

    \item \textbf{\texttt{workflow\_definitions}} -- stores workflow definitions created from the user interface, including the editable graph, compiled runtime specification, YAML rendering, validation errors, status, and timestamps.
\end{itemize}

The database stores metadata and references rather than generated runtime files. Tools are stored as references in agent configuration fields, while the actual tool code is located in \texttt{tools\_catalog/}. Prompts and schemas are also stored as URI references, while their content is resolved from MLflow. Generated runtime files, such as \texttt{agent\_config.yaml}, selected tool files, and Docker build artifacts, are created by the Agent Factory and are not persisted in PostgreSQL.

For workflow execution, the most important table is \texttt{workflow\_definitions}. It stores the editable graph in \texttt{graph\_json} and the compiled executable specification in \texttt{runtime\_spec\_json}. When the user launches a workflow, the frontend backend retrieves the saved workflow definition, validates the compiled runtime specification, and forwards it to the orchestrator through \texttt{/run-workflow-spec}.

This persistence model separates platform state from runtime execution. PostgreSQL keeps track of applications, agents, templates, workflow graphs, runtime endpoints, and configuration snapshots, while Docker handles running containers and MLflow manages prompt and schema content.

    \subsubsection{Docker-Based Deployment Implementation}

\label{sec:docker-deployment-implementation}

Docker is used to package and run each generated agent as an independent FastAPI service. In the current implementation, generated agents use the Dockerfile copied from \texttt{agent\_core/Dockerfile}. The Dockerfile is not dynamically generated; it is included in the runtime scaffold assembled by the Agent Factory.

The generated agent image uses a multi-stage build based on \texttt{python:3.11-slim}. The builder stage creates Python wheels from the generated \texttt{requirements.txt}, while the runtime stage installs these prebuilt wheels and the packages listed in \texttt{apt-runtime-packages.txt}. These dependency files are generated by \texttt{builder/services/dependency\_resolver.py} using the base runtime requirements and the dependencies required by the selected tools.

Container creation is handled by \texttt{builder/services/docker\_runtime.py}. The implementation uses Docker CLI commands to build the image, start the container, connect it to the shared Docker network, and assign a dynamic localhost host port. Each generated agent listens internally on port \texttt{8000}, and the orchestrator calls it through the shared network using an endpoint such as \texttt{http://<agent-name>:8000/run}.

After startup, the builder checks runtime readiness by calling the root endpoint \texttt{/} and the contract endpoint \texttt{/manifest}. Once the service is ready, runtime metadata such as the container identifier, container name, image tag, internal endpoint, host port, external URL, documentation URL, and runtime status is saved in PostgreSQL.

    \subsubsection{Observability Implementation}

\label{sec:observability-implementation}

The VIP platform uses Laminar as the active observability layer. TraceAI remains only as legacy code, while Laminar is initialized in the generated agent runtime and in the orchestrator. The builder itself is not directly traced, but it injects the required \texttt{LMNR\_*} environment variables into generated agent containers.

Observability is applied during agent and workflow execution. Generated agents trace \texttt{/run} calls, agent execution, tool execution, and model calls through the OpenAI-compatible instrumentation. The orchestrator traces workflow runs, dynamic workflow execution, workflow steps, and HTTP calls made to generated agents.

The main metadata captured includes the \texttt{trace\_id}, latency, token usage when available, agent name, workflow name, step index, tool name, execution status, and error information. Logs are also written to files such as \texttt{logs/app.log}, \texttt{logs/orchestrator.log}, and \texttt{logs/builder.log}, in addition to Docker and process standard output.

The frontend provides an observability interface connected to Laminar through the backend observability routes and services. This allows the user to inspect traces and execution metrics related to agents and workflows.

This implementation improves debugging and auditability by making agent execution, tool usage, workflow steps, and runtime errors visible during platform operation.

\subsubsection{Baseline Output Generation}


The second notebook implemented the baseline generation phase. It reused the golden BIAT case and the baseline prompt registered in Notebook 01, then executed the same financial analysis task with two small language models. The first model was Phi-4, accessed through an Azure OpenAI-compatible endpoint, while the second model was Llama 3.1, accessed through Groq.

The same input data and the same prompt version were used for both models. This ensured that the comparison focused on the generated outputs rather than differences in input preparation or prompt design.

The generation protocol was configured as follows:

\begin{itemize}
    \item Golden case: \texttt{test\_case\_004}, corresponding to BIAT financial data.
    \item Prompt: \texttt{financial\_analysis\_baseline}, version \texttt{11}.
    \item Models: Phi-4 and Llama 3.1.
    \item Providers: Azure OpenAI-compatible endpoint for Phi-4 and Groq for Llama 3.1.
    \item Temperatures: \texttt{0.0}, \texttt{0.1}, \texttt{0.3}, \texttt{0.5}, and \texttt{0.7}.
    \item Total outputs: ten generated reports, with five outputs per model.
    \item Maximum generation length: \texttt{1200} tokens.
\end{itemize}

Each generation call was traced with MLflow. This allowed each output to be linked to its model, provider, prompt version, temperature value, and input case. The notebook also stored execution metadata such as token usage and latency. However, latency was not used as a main comparison criterion because the two models were served through different providers and infrastructures.

At the end of the generation phase, the outputs were exported as structured files for the next notebook. The main output was \texttt{baseline\_outputs\_test\_case\_004.csv}, accompanied by a JSONL version and a compact summary file. These files contained the generated reports, the model metadata, the temperature values, token usage, and the parsed section structure.

The main artifacts produced by Notebook 02 were:

\begin{itemize}
    \item \texttt{baseline\_outputs\_test\_case\_004.csv}, containing all generated reports and metadata.
    \item \texttt{baseline\_outputs\_test\_case\_004.jsonl}, providing the same outputs in line-by-line JSON format.

    \subsubsection{Multi-Layer Evaluation Implementation}
    The third notebook implemented the evaluation phase of the SLM-Evals pipeline. It loaded the baseline outputs generated in Notebook 02 and applied several complementary evaluation layers. This multi-layer approach was necessary because financial report generation cannot be evaluated only through text similarity. A report may be fluent and close to the reference text while still containing numerical errors, missing sections, or unsupported financial conclusions.

Before computing the metrics, the generated outputs were normalized and parsed into the eight expected report sections. This allowed the notebook to evaluate both the global report and the section-level quality of each output.
    
The evaluation was implemented through the following layers:

\begin{itemize}
    \item \textbf{Deterministic similarity metrics:} ROUGE-L, BLEU, ChrF++, and BERTScore F1 were computed to compare each generated report with the reference output.
    
    \item \textbf{Structural completeness:} the notebook checked whether the eight required financial report sections were present. The structure score was computed as the ratio of detected sections to expected sections.
    
    \item \textbf{Domain numerical accuracy:} financial numbers were extracted from the generated reports and compared with the reference values using a tolerance of $\pm 2\%$.
    
    \item \textbf{RAGAS metrics:} semantic similarity, answer correctness, and factual correctness were used to evaluate semantic and factual alignment.
    
    \item \textbf{LLM-as-a-judge evaluation:} the model \texttt{openai/gpt-oss-20b} was used as the main judge, with \texttt{temperature=0.0} and \texttt{max\_tokens=3000}. It scored the generated reports on faithfulness, numerical accuracy, trend accuracy, reasoning quality, completeness, and hallucination risk. A second judge configuration, \texttt{meta-llama/llama-4-scout-17b-16e-instruct}, was also defined for additional judge or RAGAS-based evaluation support.    
    
    \item \textbf{Efficiency metadata:} token usage and execution metadata were stored, but latency was not treated as a main quality criterion because the models were served through different providers.
\end{itemize}

The main judge model used in the evaluation was \texttt{openai/gpt-oss-20b}. It was configured with a temperature of \texttt{0.0} in order to reduce randomness in the scoring process. For each generated report, the judge evaluated section-level quality according to six criteria: faithfulness, numerical accuracy, trend accuracy, reasoning quality, completeness, and hallucination risk. This judge-based layer was important because it assessed financial reasoning quality, which cannot be fully captured by deterministic similarity metrics.

The judge configuration used in Notebook 03 can be summarized as follows:

\begin{itemize}
    \item Main judge model: \texttt{openai/gpt-oss-20b}.
    \item Judge temperature: \texttt{0.0}.
    \item Maximum judge output length: \texttt{3000} tokens.
    \item Secondary judge / RAGAS judge: \texttt{meta-llama/llama-4-scout-17b-16e-instruct}.
    \item Scored dimensions: faithfulness, numerical accuracy, trend accuracy, reasoning quality, completeness, and hallucination risk.
\end{itemize}

The judge quality score was computed from the average of the first five judge dimensions, namely faithfulness, numerical accuracy, trend accuracy, reasoning quality, and completeness. The safety score was derived from the hallucination score. At section level, the final judge score was computed as follows:

\[
SectionScore = 0.80 \times QualityScore + 0.20 \times SafetyScore
\]

The final composite score combined several evaluation signals into one aggregated score:

\[
FinalScore = 0.38 \times JudgeQuality + 0.19 \times SemanticScore + 0.1425 \times Structure + 0.1425 \times Safety + 0.095 \times Efficiency + 0.05 \times ChrF
\]

In this formula, the judge quality score received the highest weight because financial report generation requires reliable reasoning, factual grounding, and controlled hallucination risk. RAGAS metrics and deterministic domain accuracy were reported as diagnostic indicators, but they were not directly weighted in the final composite score.

This evaluation design made the comparison more robust than a single-metric approach. Deterministic metrics provided a fast and reproducible baseline, structural checks verified format compliance, domain accuracy focused on financial numbers, RAGAS added semantic and factual signals, and the judge layer assessed reasoning quality. The outputs of this phase were then used to build the final leaderboard and visual comparison charts.

    \subsubsection{Results Interpretation}

   Although the detailed metric calculations are available in the evaluation notebook, the report must summarize the main findings and explain their meaning. The notebook proves that the evaluation was executed, while this subsection interprets the results obtained from the pipeline.

The aggregated comparison showed that both evaluated models were able to generate structured financial reports following the eight required sections. The structure score was equal to \texttt{1.0} for both models, which confirms that the baseline prompt successfully enforced the expected report format.
   
   The final composite score ranked Llama 3.1 slightly ahead of Phi-4, with a score of \texttt{0.648} compared with \texttt{0.643} for Phi-4. However, this difference is very small and should be interpreted carefully. The composite score includes an efficiency component, while the two models were served through different providers and infrastructures. Therefore, the final ranking should not be understood as a pure model-quality ranking.
   
   When focusing on output quality and financial correctness, Phi-4 showed stronger results on several important dimensions. It achieved better deterministic similarity scores, higher domain numerical accuracy, higher RAGAS mean score, and a higher judge quality score. This indicates that Phi-4 produced reports that were closer to the reference answer and more reliable from a financial analysis perspective.

Llama 3.1 remained a relevant candidate because it produced complete structured reports and obtained a competitive final score. However, its weaker numerical accuracy and judge quality suggest that it may require prompt improvement or additional validation before being used for more sensitive financial analysis tasks.

The main findings can be summarized as follows:

\begin{itemize}
    \item Both models respected the expected report structure, with a structure score of \texttt{1.0}.
    
    \item Llama 3.1 obtained the highest final composite score, with \texttt{0.648}, compared with \texttt{0.643} for Phi-4.
    
    \item Phi-4 obtained stronger quality-oriented results, especially in judge quality, numerical accuracy, RAGAS mean score, and reference-based similarity metrics.
    
    \item The final score difference was very small, so the result should be interpreted as a trade-off rather than a clear superiority of one model.
    
    \item Since provider latency can influence efficiency, latency was not used as the main criterion for choosing the better model.
\end{itemize}

Overall, the evaluation showed that Phi-4 was more reliable for financial report quality, while Llama 3.1 remained competitive in the global pipeline score. For a financial analysis use case, the quality-oriented metrics are more important than raw execution speed. Therefore, Phi-4 appears to be the stronger baseline for financial reasoning, while Llama 3.1 may require prompt enhancement to improve numerical accuracy and reasoning quality.

These results remain directional because the evaluation was performed on a single golden case. A larger dataset containing several banks, reporting periods, and financial situations would be required to draw stronger conclusions about the general performance of each model.


    \subsubsection{Prompt Enhancement Experiment}


After the baseline evaluation, Notebook 04 tested whether prompt engineering could improve the quality of Llama 3.1 outputs. The experiment reused the same BIAT golden case and compared the baseline Llama prompt with two variants.

The tested variants were:

\begin{itemize}
    \item \textbf{Variant A:} a Llama-optimized prompt with stricter reporting instructions and clearer trend verification rules.
    \item \textbf{Variant B:} a structured scratchpad approach where the model first extracted key financial metrics before generating the final report.
\end{itemize}

A specific check was also added for the Liquidity Coverage Ratio, noted as LCR. In this financial case, the LCR values evolved from \texttt{120} in 2021 to \texttt{125} in 2022, \texttt{130} in 2023, and \texttt{128} in 2024. The purpose of the LCR direction check was to verify whether the model correctly described this trend, instead of wrongly presenting the ratio as continuously increasing or decreasing.

The quick evaluation used ROUGE-L, ChrF++, BERTScore F1, structure score, output length, and the LCR direction check. All variants preserved the expected report structure. The baseline Llama output obtained the best ChrF++ score, while Variant A slightly improved BERTScore F1. Variant B did not outperform the baseline on automatic metrics, but it made the intermediate financial reasoning more explicit.

Although the baseline remained stronger according to the quick deterministic metrics, Variant B was registered in MLflow as an optimized prompt for future experimentation. The registered prompt was named \texttt{financial\_analysis\_llama\_optimized}, version \texttt{1}, with the alias \texttt{llama\_current}.

This experiment should therefore be interpreted as exploratory. It showed that prompt design can influence the generated report, but more golden cases are required before concluding that one prompt variant is consistently better.
